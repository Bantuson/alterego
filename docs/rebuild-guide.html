0 · The six thinking moves that happen before any code

You said your weakness is "thinking in a way that even allows me to start writing code." Here is the honest answer: nobody starts by writing code. The start is a set of compressions — turning a fuzzy wish into shapes small enough to hold in your head. These are the six moves made for alterego, in order, before a single line existed. They transfer to any project.

Move 1 — Compress the vision into a pipeline sentence.
The brief was huge: record, look different, sound different, fix lighting, edit, brand, post. Compression: "record → disguise → enhance → cut → publish." If you cannot draw your project as boxes and arrows in one line, you don't understand it yet — keep compressing before you code. Every noun in that sentence later became a Python module.
Move 2 — Choose the interface between parts before the parts.
Decision: every stage reads a video file and writes a new video file. This one choice quietly solved five future problems: stages can run in any order, be re-run alone, be inspected between steps, be tested independently, and a crash loses nothing. When stuck on "where do I start", start by asking "what do the pieces hand each other?"
Move 3 — Let the constraint design for you.
4 GB of RAM is not an obstacle, it is an architect. It dictated: stream one frame at a time (never load a video), prefer classical CV over deep nets, lean on ffmpeg's C code for everything media-shaped, compute expensive fields at quarter resolution. Strong constraints REMOVE decisions — that's a gift.
Move 4 — Build the riskiest thing first.
The face disguise was the make-or-break: if "geometric warp = different-looking person" was wrong, the whole project was dead. So it was built and previewed on day one, before comforts like a nice CLI. Ask: "which part, if impossible, kills the project?" Build exactly that first.
Move 5 — Decide how you'll verify before you build.
There was no test footage and no reviewer. So every claim got a synthetic verification designed up front: a 440 Hz tone to prove pitch math by FFT, a tone/silence pattern to prove the cutter's arithmetic, text-to-speech saying "um" to prove filler removal, a brightness histogram to prove why night footage fails. If you can't say how you'd prove a piece works, you don't know what "works" means yet — and that ambiguity is what makes starting feel hard.
Move 6 — One idea per commit.
36 commits, each one nameable in a sentence. This is not bureaucracy: it forces you to know what you are doing before doing it (you can't name a commit you don't understand), and it turns the git log into the project's autobiography — which is literally the spine of this document.

2 · Don't rewrite what already exists (ffmpeg)

46fdc0c feat: ffmpeg helpers and terminal recorder

"Which parts of this problem are already solved by professionals?"

Reading/writing compressed video+audio is a decades-deep problem. The professional move is to keep Python for the interesting part (per-frame vision) and delegate encoding, devices, containers, and cutting to ffmpeg subprocesses. This layer is two ideas: find ffmpeg robustly (PATH → winget's link folder → the pip-bundled fallback — ordered by capability) and run it safely. One non-obvious craft detail: ffmpeg prints analysis output to stderr, and some commands "fail" by design (listing devices exits non-zero) — so there is a probe_stderr() helper that deliberately ignores exit codes. Reading a tool's quirks IS the work.
def _stop_ffmpeg(proc):
    # Sending "q" to ffmpeg's stdin is the clean way to stop: it
    # finishes writing the file header so the video stays playable.
    # Killing the process instead can corrupt the recording.
    proc.stdin.write(b"q")

3 · The 4 GB trick: one frame at a time

c3f3758 feat: streaming video processor and landmark wrapper

"What is the one loop every video stage shares?"

Do the RAM math out loud: one 720p frame ≈ 2.6 MB as a numpy array; a 5-minute video ≈ 24,000 frames ≈ 60 GB if loaded whole. Read-one, process-one, write-one needs ~3 MB. That math is the design. Then notice: disguise and enhance differ only in the function applied per frame — so the loop is written once, taking a frame_fn parameter. This is the single most reusable abstraction in the codebase, and it fell out of asking "what do these two future modules have in common?"
# A frame function takes a BGR image and returns a new one.
FrameFn = Callable[[np.ndarray], np.ndarray]

def stream_video(src, dst, frame_fn: FrameFn) -> None:
    while True:
        ok, frame = capture.read()
        if not ok: break
        writer.write(frame_fn(frame))   # the ONLY line that varies
    remux_audio(temp, src, dst)         # OpenCV drops audio; ffmpeg puts it back

4 · Eyes: landmarks and why they jitter

"How does code know where a jaw is?"

MediaPipe Face Mesh returns 468 numbered points, and index 152 is always the chin — anatomical addressing. That stability is the foundation the disguise stands on. But detection jitters a pixel or two per frame, and warping with raw landmarks makes the whole face shimmer. Fix: an exponential moving average — keep α of the new position, 1−α of the running average. Two design details worth defending: the smoother resets on lost faces (blending positions across a gap would smear stale geometry onto the returning face), and it lives in its own class because smoothing-over-time is a separate concern from detecting-in-a-frame.
class LandmarkSmoother:
    def update(self, landmarks):
        if landmarks is None:
            self._state = None          # face lost: never blend stale positions
            return None
        self._state = (self.alpha * landmarks
                       + (1 - self.alpha) * self._state)
        return self._state

5 · The core bet: identity is geometry

99258b6 feat: seeded geometric face disguise via Gaussian displacement fields

"What makes a face recognizable — and what is the smallest change that breaks it?"

The domain insight: recognition (human and machine) keys on ratios — jaw width to face width, eye spacing, nose length. Shift each by a few percent, consistently, and the face reads as a different person while staying perfectly natural. That translates to three sub-problems, solved in order:

(a) A reproducible identity. Eight numbers drawn from a seeded RNG. Same seed → same face, forever. The seed IS the alter ego — which later forced two more decisions (persist it to a file; gitignore that file like a password).

(b) Semantic shifts. A table maps profile knobs to landmark motions: "jaw wider" = push jaw points away from the face's center line, scaled by measured face width so leaning into the camera doesn't break it.

(c) A smooth warp. Each control point drags nearby pixels with Gaussian falloff (think Photoshop's Liquify). Summing per-pixel for 720p is too slow — so the field is computed at quarter resolution and upscaled, which is visually lossless because smooth fields stay smooth when upscaled. That trick (compute small, upscale) recurs three more times in this codebase.
# remap() is a BACKWARD warp: each output pixel asks "which input
# pixel do I show?". To push a feature +d we sample from -d,
# hence the minus sign on the shifts.
field_x, field_y = displacement_field(shape, points, -shifts, sigma)
Same seed twice → identical profile (the privacy promise as an assert). Zero strength → zero shifts. Max shift ≤ 6% of face width ("naturally different" quantified). The field peaks at a control point and dies far away. All pure math — no camera, no model, millisecond-fast.

6 · Cheap classical wins: grading and cutting

8e9f27c feat: lighting/color enhancement and silence-cutting stages

"Which named, googleable techniques fix 80% of amateur footage?"

Deliberately boring, deliberately classical — each step is a technique with a name you can look up: gray-world white balance (if the channels' averages disagree, the light is tinted, not the scene), CLAHE (local contrast: lifts shadows without washing highlights), gentle saturation. Order matters and is worth defending: balance color first so CLAHE works on true tones; saturate last to restore what correction flattened.

The cutter is a different lesson: most media programming is interval arithmetic. ffmpeg's silencedetect prints silence timestamps; the real work is inverting "silence ranges" into "keep ranges" with padding — pure list logic. The padding (0.15 s of breath kept at every cut) is the difference between an edit that feels human and one that feels robotic. Pacing is padding.
def keep_segments(silences, duration, padding=0.15):
    kept, cursor = [], 0.0
    for silence_start, silence_end in silences:
        keep_end = min(silence_start + padding, duration)
        if keep_end - cursor > 0.05:
            kept.append((cursor, keep_end))
        cursor = max(silence_end - padding, cursor)
    ...
No silences → keep everything. One middle silence → two segments. Silence at the start/end → dropped. Kept time never exceeds duration; segments never overlap. Notice: these tests describe the function's contract in English first — write those sentences, then translate to asserts.

7 · A CLI shaped like git

b68eb36 feat: argparse CLI with live disguise preview mode ada450a test: cover disguise math and silence-cut interval logic

"What's the smallest interface that makes every module usable?"

One subcommand per pipeline stage — the same pattern as git, uv, and docker, built with stdlib argparse (zero dependencies, transferable knowledge). Two touches matter: modules are imported inside command functions so alterego --help doesn't pay MediaPipe's import cost, and outputs default to traceable names (take.mp4 → take_alterego.mp4) so a folder listing tells the story of what was done. The preview command (press N for a new random face, K to keep it) exists because a feedback loop beats a spec — you choose your alter ego by seeing it, not by imagining it.

8 · Reality fights back (the fix commits)

33cac35 fix: migrate to MediaPipe tasks API a7c9b86 fix: record at camera-native mode; fail fast acf7398 feat: persist chosen alter-ego seed

"What did running it teach that reading it couldn't?"

Three collisions with reality, three lessons:

The API that vanished. The code was written against MediaPipe's classic mp.solutions API; version 0.10.35 had deleted it. Diagnosis method: print(dir(mp)) — ask the library what it actually has, never argue with it. The rewrite (tasks API) also surfaced a hidden requirement: the model file ships separately, so add a download-once-and-cache step.

The camera that said no. Recording failed with "Could not set video options" — the code demanded 1280×720@30 and the hardware's menu only goes to 640×480. Lesson burned in: never force hardware parameters; ask the device what it supports (devices --modes), default to native. The fix also added a fail-fast: poll capture processes 2 s after start, because a doomed recording should die loudly now, not silently after a 20-minute take.

The seed that lived in scrollback. The user pressed N twice in preview and asked "did that persist anywhere?" It hadn't — the identity existed only in terminal history. When a value IS the product (the seed is the alter ego), it gets a file, and that file gets gitignored like a credential.

9 · Growing sideways: background, fillers, voice

846940d feat: background stage 0b2dabc feat: disguise reports face coverage d7d0652 feat: cut merges filler ranges with silences e5803f0 feat: seeded voice pitch shift f2d3f40 feat: ship command

"Now that the spine works, what does each new limb reuse?"

Every extension deliberately rides existing rails — that's what a good foundation buys:

Background replacement = a person-mask model + the same stream_video loop + one new pure function (composite). The realism rules are craft knowledge worth stating: blur the fake backdrop (sharp = green-screen; blurred = "portrait mode"), and grade after compositing so subject and scene share one color treatment.

The coverage warning came from a measurement that changed the project's soul: on a dark take, faces were detectable on 2% of frames — meaning the "disguised" output was 98% the user's real face, silently. Silent privacy failure is the worst failure. Now disguise counts coverage and shouts below 90%.

Filler removal is the reuse masterclass: the cutter already removes time ranges; fillers are just more ranges, sourced from whisper word timestamps. One trap: the cutter keeps 0.15 s of padding around every cut (good for pauses), which would let a 0.3 s "um" escape whole — so filler ranges are pre-expanded by exactly that padding. When two features share a mechanism, check where their needs differ.

Voice mirrors the face philosophically: factor derived from the same seed (consistent forever), but salted so voice and face are independent draws, and the sampling designs out the do-nothing zone around 1.0. Ship then chains everything, ordered by reasons, not habit: disguise first (detection wants raw pixels), grade after background (glue), cut before voice (whisper should hear the natural voice).
The physics test for voice: synthesize 440 Hz, shift ×1.10, FFT the output → measured 485 Hz (ratio 1.1023). When your code claims to change a physical quantity, measure the quantity.

11 · Live mode: the safety model inverts

5ff286f feat: real-time pitch shifter (two-tap delay line, fail-closed) b58cc05 feat: live vision pipeline - cached warps, Nth-frame masks, fail-closed

"What changes when there is no 'post' — when the frame ships the instant it exists?"

In post, a frame with no detected face passes through untouched, and you review before publishing. Live, that same behavior would leak your REAL face to an audience, unreviewably. The 2%-coverage measurement from layer 9 is what makes this concrete rather than theoretical. So live mode fails closed: landmarks lost → the person region gets pixelated (we measured that segmentation survives darkness that kills landmark detection — the fallback is built on evidence, not hope). Pixelation over blur is a UX decision: it reads as deliberate censorship, not a broken camera. The voice thread obeys the same law: any error in the shifter outputs silence, never the raw microphone. When output is irreversible, every failure path must be designed, not defaulted.
Real-time also changes the algorithms. File-based pitch shifting hands ffmpeg a whole file; live audio arrives in 23 ms blocks. The solution is the two-tap delay-line shifter (the guitar "Whammy pedal" trick): a ring buffer, two read heads moving at a different speed than the writer, crossfaded so each head's jump lands at zero gain. And the vision loop exploits time-coherence: bodies barely move in 100 ms (segment every 3rd frame), faces barely move between frames (rebuild warp maps only when landmarks moved > 1.5 px — below that it's detection jitter, and the cached warp is more stable).
The pipeline takes its models as constructor arguments, so tests inject fakes that return scripted landmarks — including None sequences that prove the fail-closed path fires and the output frame is measurably scrambled. The shifter is verified by the same FFT method as layer 9, fed through live-sized blocks.

The three testing layers

alterego has 45 tests and none of them need a camera, a GPU, or your face. That is not an accident — it's a doctrine with three layers:

Layer 1 — Pure logic, synthetic inputs

Separate the decision from the I/O, then test decisions with made-up arrays: fake landmarks in a box, masks of ones and zeros, silence lists like [(4.0, 6.0)]. This is why functions like keep_segments, control_shifts, and filler_ranges take plain data instead of reading files — testability is an input to the design, not an afterthought.

Layer 2 — Physics checks

When code claims to change a measurable quantity, measure it: pitch shift ×1.10 → FFT peak moves 440 → 485 Hz. Displacement field → full shift at the control point, zero far away. Night mode → mean brightness 18.8 → 56. The test doesn't trust the code's story; it measures the world the code changed.

Layer 3 — End-to-end with synthesized media

No real footage needed: ffmpeg's testsrc2 generates video, a math expression generates tone/silence audio with known timings, and Windows TTS literally speaks "um" so filler removal can be proven to cut exactly 3 fillers. Fabricating inputs with known answers is a superpower — you always know what the output should be.

Why never test with your own face? Privacy (test artifacts get committed, shared, pasted into issues) and reproducibility (your face can't be regenerated from a seed on someone else's machine; testsrc2 can).

Defending it: what a senior will ask

Q: Why geometric warping instead of a deep-learning face swap?
Three reasons, in order: it runs on a 4 GB CPU machine (deepfakes don't); pixels only move and are never invented, so lip-sync, skin texture, and expressions survive perfectly; and it's fully deterministic from a seed, which is what makes the identity reproducible across every video. The trade-off accepted: it raises the cost of recognition rather than providing cryptographic anonymity — stated honestly in the README.
Q: Why files between stages instead of one in-memory pipeline?
Inspectability (watch the output of any stage), resumability (a crash in stage 4 keeps stages 1–3), independent re-runs, trivially testable stages, and constant memory. The cost — re-encoding between stages — is paid in disk and CPU time, which this project has, instead of RAM, which it doesn't. Live mode is the exception that proves the rule: there the deadline dominates, so it uses one in-memory loop with caching.
Q: Why is the seed treated like a password?
The profile is derived from it deterministically; anyone with the seed can reproduce the exact warp — or approximately invert it. It also IS the brand: a different seed is a different face on camera. Hence: stored in a file (terminal scrollback is not storage), gitignored, never in commit history.
Q: Your displacement field just sums Gaussians. Why not thin-plate splines or piecewise-affine triangulation?
Considered and rejected for v1: TPS needs opencv-contrib and is harder to reason about; triangulation is the classic approach but ~3× the code and its artifacts (visible triangle seams) are worse than Gaussian falloff's (slight neighborhood drag) at these tiny shift magnitudes (≤6% of face width). The Gaussian field is 40 lines, vectorized, and computed at ¼ resolution for speed. At larger deformations the answer would change.
Q: How do you know the disguise actually protects anything?
Claim is scoped honestly: it shifts the geometric ratios that recognition keys on, consistently. What's measured: same seed → identical profile (tested), shifts ≤6% face width (tested), coverage reported per run (the 2% night-footage incident is why). What's NOT claimed: robustness against determined forensic analysis — voice content and context identify you more than biometrics, which is why voice shifting exists and why the README says "raises the cost, not cryptographic anonymity."
Q: Why does live mode pixelate instead of just skipping frames when the face is lost?
Skipping (freezing) looks like a broken stream and still shows the last real-ish frame; passthrough leaks the real face. Pixelation is fail-closed (nothing identifiable leaves), cheap (downscale + nearest-neighbor upscale), and reads as deliberate to viewers. The person region comes from segmentation, which we measured to survive lighting that kills landmark detection — the fallback is built on the failure mode's actual shape.
Q: Whisper tiny is a weak model. Why is it enough here?
Task-model fit: filler detection only needs the acoustically distinct easy words ("um", "uh") plus timestamps; caption text quality is nice-to-have, not correctness-critical. tiny+int8 fits in <1 GB RAM, which is the binding constraint. Verified end-to-end: it caught 3/3 scripted fillers in synthesized TTS speech — a harder acoustic target than natural fillers.
Q: What's the scaling story?
Vertical: the same code hits 1080p30 on any machine with a real GPU — nothing is architecture-bound to 480p, the resolutions are parameters. Horizontal: heavy offline jobs (Remotion renders, future RVC voice conversion) are designed to move to free-tier cloud (GitHub Actions' 2,000 min/month, Colab) because every stage is a file-in/file-out CLI command — trivially portable.
Q: Where are the known weaknesses?
Being able to answer this is what makes the rest credible: (1) the warp is axis-aligned relative to the face — strong head rotation reduces disguise strength; (2) the delay-line pitch shifter adds a faint shimmer at large factors; (3) background edges show halo under hair in high contrast; (4) live mode depends on OBS's driver for the virtual camera; (5) the voice shift changes pitch, not vocal-tract character — RVC-grade conversion is roadmapped for Colab.
